# PLAN.md — Воркфлоу разработки KAZKIOTI

> **Документ для главного агента.** Описывает правила делегирования, фазы работы и точки контроля.
> Главный агент **не пишет код сам** — он читает входные документы, декомпозирует задачи, делегирует субагентам и сводит результаты.

---

## 0. Контекст проекта (минимум, который должен знать каждый агент)

- **Заказчик:** KAZKIOTI — казахстанский завод сельхозтехники (тракторы 40–210 л.с. + навесное).
- **Задача:** новый сайт взамен текущего kazkioti.kz.
- **Языки:** KZ + RU (английский отложен).
- **Скоуп страниц (v1):** `/`, `/tractors`, `/tractors/[model]`, `/tractors/compare`, `/attachments`, `/parts`, `/dealers`, `/cases`, `/cases/[slug]`, `/news`, `/news/[slug]`, `/about`, `/faq`, `/contacts`.
- **Источники истины:**
  - [docs/kazkioti.md](docs/kazkioti.md) — компания, модели, скоуп.
  - [docs/DESIGN.md](docs/DESIGN.md) — ТЗ на дизайн.
  - [docs/kazkioti-handoff.zip](docs/kazkioti-handoff.zip) — handoff от Claude Design: HTML/CSS/JS-прототипы 14 страниц + `theme.css` (Direction **B — Editorial-Minimal**), `base.css`, `app.js`, hero-видео, фото DF 904 / TS 1404.
- **Дизайн-фикс:** Editorial-Minimal, Space Grotesk + Inter + JetBrains Mono, палитра белый + графитовый + brand-red `#e0001b` + brand-blue `#1853d6`.
- **Папки `raw/` и `uploads/`** — сырьё, в `.gitignore`, агенты их не читают.

## 1. Правила делегирования (главное)

> **Принцип:** главный агент координирует, субагенты исполняют. Главный агент не редактирует код руками. Если задача попадает в одну из категорий ниже — она уходит субагенту.

### 1.1. Когда какой агент

| Тип задачи                                                                 | Кому делегировать          | Модель | Почему                                              |
|----------------------------------------------------------------------------|----------------------------|--------|-----------------------------------------------------|
| Связные задачи (несколько шагов одной фичи, общий контекст)                | **один Opus-агент**        | Opus   | Контекст не дробится, агент держит фичу целиком.    |
| Сложная работа (архитектура, многофайловая реализация, нетривиальная логика) | **один Opus-агент**        | Opus   | Нужна максимальная глубина рассуждения.             |
| Небольшая изолированная задача (правка стилей, добавление компонента, мелкий рефактор) | **Sonnet-агент**           | Sonnet | Быстро, дёшево, достаточно качества.                |
| Исследование/поиск по кодовой базе или докам                               | **Explore-агент**          | —      | Read-only, экономит контекст главного.              |
| Архитектурный план перед большой работой                                   | **feature-dev:code-architect** | Opus   | Спецагент с фокусом на blueprint.                   |
| Параллельное исследование нескольких независимых вопросов                  | **несколько Explore** в одном сообщении | —      | Запускать параллельно через несколько Agent-вызовов одним сообщением. |

### 1.2. Ревью (строго)

Для ревью только два варианта:
- **Codex (`codex:rescue`)** — second opinion, независимый взгляд снаружи.
- **Opus (через `feature-dev:code-reviewer` или прямой Agent с subagent_type=feature-dev:code-reviewer)** — main reviewer.

Никаких Sonnet-ревью. На крупных мерджах допустимо запускать **Codex + Opus параллельно** (одним сообщением, два Agent-вызова) и потом сводить замечания.

### 1.3. Запрещено главному агенту

- Писать/редактировать код напрямую (Edit/Write/NotebookEdit) — всегда через субагента.
- Делать массовые поиски через Grep/Glob, если по объёму это тянет на отдельную задачу — это Explore.
- Делать ревью самому — это Codex или Opus-ревьюер.

Исключения, когда главный действует руками:
- Создание/правка `PLAN.md`, `CLAUDE.md`, `MEMORY.md`, `.gitignore` и других мета-файлов воркфлоу.
- Чтение результатов агентов, сведение, делегирование следующего шага.
- Точечный ответ пользователю на вопрос, не требующий работы с кодом.

### 1.4. Параллелизм

Если задачи независимы — запускать агентов в **одном сообщении несколькими Agent-вызовами**. Примеры:
- Два Explore-агента изучают `tractor-df904.html` и `index.html` одновременно.
- Codex и Opus-ревьюер ревьюят один PR параллельно.
- Sonnet-агент верстает footer, пока Opus-агент собирает hero.

## 2. Технологический стек (зафиксирован 2026-05-20)

- **Framework:** Next.js (App Router) + TypeScript + Tailwind CSS.
- **Deploy:** Vercel (preview на каждый PR, prod на `main`).
- **CMS v1:** MDX-в-репо (`/content/news/*.mdx`, `/content/cases/*.mdx`, `/content/faq/*.mdx`, `/content/tractors/*.mdx`). Возможна миграция на Sanity/Strapi позже.
- **i18n:** `next-intl` или `next-i18next` — конкретику определяет архитектор в Фазе 2.
- **Styling:** Tailwind theme отражает токены из `.handoff/kazkioti/project/theme.css`.

## 3. Фазы работы

### Фаза 1 — Подготовка проекта (главный агент сам, без кода)

1. Создать `CLAUDE.md`, `PLAN.md` (этот файл), `MEMORY.md`, `.gitignore` (`raw/`, `uploads/`, `node_modules/`, `.next/`, `dist/`).
2. Распаковать `docs/kazkioti-handoff.zip` во временную папку **только для чтения агентами** — не коммитить распакованное.
3. Зафиксировать стек (вопрос заказчику).
4. Инициализировать git-репозиторий и первый коммит «chore: bootstrap docs and workflow».

**Выход фазы:** репо с документами, `.gitignore`, согласованный стек.

---

### Фаза 2 — Архитектура кода (один Opus-агент: `feature-dev:code-architect`)

Делегировать одному Opus-агенту:
- Изучить handoff (`index.html`, `theme.css`, `base.css`, `app.js`) и `DESIGN.md`.
- Спроектировать структуру проекта под выбранный стек: папки, маршруты, дизайн-токены (CSS-переменные или Tailwind theme), компонентная библиотека, i18n-стратегия, источник данных (моки/MDX/CMS).
- Выдать blueprint с конкретными файлами для создания и порядком сборки.

**Выход фазы:** `ARCHITECTURE.md` (в корне) + список задач для Фаз 3–4.

---

### Фаза 3 — Каркас и дизайн-система (один Opus-агент)

Один Opus-агент собирает фундамент:
- Инициализирует проект (Next/Astro/...), Tailwind, шрифты, токены из `theme.css`.
- Реализует `layout` (Header sticky, Footer, FAB мессенджеров, Cookie-баннер).
- Реализует i18n (KZ/RU) с переключателем.
- Реализует компонентную библиотеку: `Button`, `Input`, `Textarea`, `Select`, `PhoneInput`, `Card (tractor/news/case)`, `SpecTable`, `Accordion`, `Tabs`, `Modal/Sheet`, `Toast`, `Pagination`, `Breadcrumbs`, `CompareTray`, `VideoPlayer`, `MapPin`.
- Реализует базовый `MiniContactForm` (имя + телефон).

**Выход фазы:** работающий каркас с пустыми страницами и собранным storybook-подобным indexом компонентов (если решено) или просто `/_components` маршрутом для визуальной проверки.

**Ревью:** Codex + Opus-ревьюер параллельно (один проход).

---

### Фаза 4 — Страницы (по одному Opus-агенту на связную группу)

Разбивка по связности (внутри группы — один Opus-агент держит контекст):

- **Группа A — Главная + Каталог + Карточка трактора + Compare**
  Самая важная связка. Один Opus-агент.
- **Группа B — Навесное + Запчасти/сервис + Дилеры**
  Делит карту/сервисные паттерны. Один Opus-агент.
- **Группа C — Новости + Кейсы (+ деталки)**
  Шаблон списка/статьи общий. Один Opus-агент.
- **Группа D — О компании + FAQ + Контакты**
  Статика. Можно отдать **Sonnet-агенту** (задача не сложная, паттерны уже есть из Фазы 3).

Каждая группа = отдельный Agent-вызов. Если группы независимы по файлам — **запускать параллельно одним сообщением** (несколько Agent-вызовов).

**Ревью после каждой группы:** Codex (second opinion) + Opus-ревьюер. На мелких — допустим только Opus-ревьюер.

---

### Фаза 5 — Контент и моки

- Sonnet-агент заводит моки: 6 моделей тракторов, 6+ навесных, 3 кейса, 6 новостей, FAQ-структура. Источник — `docs/kazkioti.md` (раздел 4 — модели, раздел 5 — навесное).
- Те поля, которых нет в `kazkioti.md` (TT 1004, фото некоторых моделей, реальные дилеры) — отметить как `TBD` и собрать в один список `docs/CONTENT_GAPS.md` для заказчика.

---

### Фаза 6 — Motion, видео, a11y, performance

Один Opus-агент проходит по чеклисту из `DESIGN.md` §7–§11:
- Scroll-reveal, count-up, hover-loop на карточках, Compare-tray slide-up, Ken Burns на hero.
- `prefers-reduced-motion` поддержка.
- Lazy-load видео + постеры, `loading="lazy"` на изображениях.
- Контраст AA/AAA, focus-ring, aria-label на иконках.
- Lighthouse: цель ≥ 90 на Performance/Accessibility/SEO для главной и карточки трактора.

**Ревью:** Codex (производительность/a11y) + Opus-ревьюер (UX/корректность motion).

- `/ultrareview` (по запросу пользователя) для финальной оценки.

## 4. Точки контроля главного агента

После каждой фазы главный агент **сам** (без субагента):
1. Прочитывает результат, сверяет с `DESIGN.md` и `kazkioti.md`.
2. Если есть отклонения — формулирует точечную задачу и делегирует одному агенту (Sonnet, если правка мелкая; Opus, если затрагивает архитектуру).
3. Обновляет `MEMORY.md` и/или `PLAN.md`, если по ходу всплыли решения, которые должны пережить сессию.
4. Коммит после успешного ревью (через Sonnet-агент с одной задачей «прокоммить» — главный не запускает git сам, если правит код субагент).

## 5. Чек-лист для каждого делегирования

Перед запуском Agent-вызова главный агент в промпте субагенту обязательно даёт:

- [ ] Чёткая цель задачи (в одной фразе).
- [ ] Ссылки на источники истины (`docs/kazkioti.md`, `docs/DESIGN.md`, конкретный HTML из handoff).
- [ ] Файлы, которые нужно создать/изменить (если известно).
- [ ] Что **не** трогать (границы скоупа).
- [ ] Формат отчёта (что вернуть главному: список изменённых файлов, ключевые решения, известные TODO).
- [ ] Указание следовать `CLAUDE.md`.

---

## 6. Антипаттерны (чего избегаем)

- Запуск Opus-агента на тривиальную правку (лишний расход).
- Запуск Sonnet-агента на архитектурное решение (риск поверхностности).
- Ревью силами Sonnet — запрещено.
- Главный агент пишет код в обход субагента — запрещено.
- Несколько агентов правят пересекающиеся файлы одновременно — конфликты. Параллелим только по непересекающимся областям.
- Делегирование без ссылок на `docs/` — субагент додумает и уйдёт не туда.
